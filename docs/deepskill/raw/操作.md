對，**我會強烈傾向讓 Volva 用戶只跟一個主代理聊天**。
不是因為後面真的只有一個 agent，而是因為：

> **前台應該只有一個連續人格 / 接洽面；後台才是多容器、多技能、多 worker。**

不然一上來就把多代理直接暴露給使用者，體驗很快就碎掉。

# 我的答案先講白

## 是，基本形態很像：
- **前台**：一個主代理跟用戶對話
- **後台**：把 skill、workers、dispatch、world、review 這些東西當成手腳與工作單位
- **旁路**：另外一個 control / monitoring 面板負責看進度、介入、終止、批准

但我會補一句很重要的：

> **主代理不只是 orchestrator。**
> 它更像 **host / editor / liaison / steward**。

也就是它做的不只是分配工作，還包括：
- 理解你到底在做什麼
- 幫你選容器
- 維持語境連續性
- 把後台結果翻回人話
- 幫你在「探索 / 執行 / 收成」之間切換

---

# 所以我會把整體切成三個面，而不是一坨

## 1. Conversation Plane
也就是使用者真正接觸到的那個主代理。

這個代理可以是：
- 接洽助理
- 某個角色人格
- 世界裡的 steward / host
- 一個比較中性的工作代理

但無論怎樣，**對使用者最好是單入口**。

它的工作是：
- 接收需求
- 判斷工作姿態
- 選容器
- 決定要不要叫 skill / task / review / world
- 回報進度
- 請求人類批准
- 收斂結果

這是前台。

---

## 2. Execution Plane
這裡才是後台真正做事的地方。

它包含：
- skills
- task workers
- review/investigation workers
- Karvi dispatch
- world actors / room actors
- runtime actions

這裡可以很多 agent，
甚至可以很多 repo、很多 runtime、很多 session。

但它們不該直接變成用戶介面。

---

## 3. Control Plane
這是你剛問到的：
- 監看進度
- 中止
- 暫停
- 改方向
- 重新派工
- approve / reject

我會把這個獨立成一個 **旁路面**，而不是把所有控制都塞進主聊天流。

原因很簡單：

如果所有東西都塞進聊天，你很快會遇到：
- 訊息被工作 log 淹沒
- 人類只是想停一下，卻要進對話上下文解釋半天
- 主代理一邊聊、一邊回報、一邊吐內部狀態，會很吵

所以控制面最好是**獨立存在，但可由主代理代操作**。

也就是 **控制指令**，不該和普通任務語義混在一起。

- 轉譯結果

# 我會更具體地切成四種使用狀態

## A. Chat mode
你只是在跟主代理討論：
- 想法
- 問題
- 方向
- 怎麼處理

這時它主要在：
- 理解
- 選容器
- 不一定做很多後台動作

---

## B. Work mode
主代理已經替你啟動某個容器：
- Skill
- Task
- Review
- World process

這時後台開始跑。

主代理只需要回你：
- 已啟動什麼
- 接下來做什麼
- 哪裡需要你
- 什麼時候再回來

---

## C. Watch mode
你想看進度，不想重跑對話。

這時應該有單獨的東西讓你看：
- 現在跑到哪
- 哪些 worker 在做事
- 哪些已完成
- 哪些卡住
- 有沒有風險/等待批准

這種東西最好是 timeline / board / status，不是聊天泡泡。

---

## D. Control mode
你要干預：
- 暫停
- 終止
- 重試
- 改目標
- 限縮範圍
- 批准下一步

這是控制操作，不是一般對話內容。

## 使用者感覺上：
我只是在跟一個主代理互動。

## 系統實際上：
- 主代理在背後選容器
- 啟動 skill 或 workers
- 透過 Karvi 派工
- 在 Thyra runtime 裡受約束
- 把重要事件寫進 Edda
- 並且隨時可經 control plane 被監看 / 暫停 / 終止

這才是比較穩的設計。

# 那主代理要不要角色扮演？

## 可以，但我會把角色扮演當成 **frontstage skin**，不是底層架構。

也就是：

- 對某個世界，主代理可以是一個 steward / mayor / host
- 對某個工作場景，它也可以是接洽助理
- 對某個創作場景，它甚至可以是角色化代理

但底層都還是：
- 容器選擇
- 技能調用
- 派工
- 控制
- 記憶

所以角色是表面層，不該主導系統設計。

# 我也會提醒一個風險

## 不要讓主代理「假裝全都自己做」
這在早期很容易發生，因為聊天看起來比較流暢。

但後果是：
- 沒有清楚的 work object
- 沒有可監看的 execution
- 沒有可中止的單位
- 沒有可回收的 artifact
- 一切都埋在對話裡

這樣很快就不能 scale。

所以主代理應該會說：
- 我幫你開了一個 review run
- 我把這段交給 deploy skill 處理
- 現在有兩個 worker 在後台跑
- 這一步要你批准
- 要不要看進度/停止

也就是要讓使用者知道後台真的有「工作單位」存在。

# 1. 先定產品核心：Volva 到底是什麼

這四件事，我覺得就是 Volva 的產品骨架。

# 4. Volva 的真正價值，不在 mode 本身，而在 routing

因為同一句話：

- 有時是 Review
- 有時甚至該先 Shape

決定因素不是句子本身，

- 風險

所以我會做成：

## **container routing protocol**
不是：
- 把一句話分到 label

- 看目前工作狀態，先選主容器，再允許轉移
