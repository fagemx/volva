# 先回答第一題：另外三個容器可能是什麼？

你現在已有三個：

1. **Village / World container**
持續性的工作世界、房間、角色、shared state

2. **Intent-router / Space-builder container**
問題還沒定型，先判路、拆 realization space、找 regime

3. **Skill container**
已成熟、可重用、可治理的方法包

如果要補成 6 個，我覺得最自然的另外三個是：

---

## 4. **Task / Dispatch container**
這個是：

> **有明確輸出、邊界清楚，但不一定值得 skill 化的單次工作容器**

適合：
- 一次性 bounded task
- 有明確 deliverable
- 直接派工就能做
- 不需要先建世界，也不需要先抽成 skill

例子：
- 改某個 function
- 補測試
- 把一份文檔整理好
- 跑一個 repo fix
- 做一次 specific migration

這個很重要，因為不然所有事都會被迫：
- 不是進 skill
- 就是進 router

那會太重。

---

## 5. **Review / Investigation container**
這個是：

> **先查清楚、判斷、比對、審視，而不是立刻做事的容器**

適合：
- debug / oncall
- code review
- “這個架構哪裡有洞？”
- “這份 spec 值不值得 promotion？”
- “這次 rollout 為什麼 fail？”

它跟 router 不一樣：

- **router** 是前處理：還不知道怎麼處理這題
- **review/investigation** 是 evidence-first：知道要查什麼類問題，但要先看證據、再判斷

這個容器很值錢，因為很多高階工作不是執行，而是判讀。

---

## 6. **Harvest / Promotion container**
這個是：

> **把一次性做法、反覆經驗、真實失敗，收斂成可重用能力/規則/skill 的容器**

適合：
- “把剛剛那套流程收起來”
- “這類問題之後常會遇到，幫我做成 skill”
- “把這次 incident 的教訓升成 gotcha / policy”
- “把這個 ad hoc run 變成可重用模板”

這個容器也很關鍵，因為沒有它：
- 任務做完就蒸發
- skill 不會從真實世界長出來
- Edda 也沒有地方被回收進方法層

# 為什麼分類器容易爛？

因為它預設這些類別是：

- 固定
- 互斥
- 能從單句語義直接判出來

但實際上都不是。

---

## 問題 1：同一句話，依上下文會進不同容器

例如：

> 「幫我 deploy checkout-service」

可能是：

- **Skill**：如果已有成熟 deploy skill
- **Task**：如果只是一次性 staging deploy
- **Review**：如果其實是 deploy fail 後要查原因
- **Harvest**：如果你剛跑完 3 次，想把流程收成能力

所以容器不是從文字單獨決定，
而是從：

- 現有 skill 庫
- world 狀態
- 風險級別
- 過去 precedent
- 這次工作是“做”還是“查”還是“收”

一起決定。

---

## 問題 2：很多請求不是一個容器，而是容器序列

例如：

> 「我想把這個產品方向想清楚，然後做成可重用流程」

這不是單一 label。
可能是：

**Shape → World → Harvest**

或者：

**Review → Skill → Task**

所以 container selection 本質上比較像：

> **先選 primary container，再允許後續 transition。**

不是一次定生死。

---

## 問題 3：容器的正確落點跟「你現在有什麼能力」有關
這點很重要。

同一題如果：
- 你已經有成熟 skill → 進 Skill
- 你還沒有 skill → 進 Task 或 Shape
- 你有長期 project world → 進 World 裡處理

所以 selection 不是純語意分類，
而是：

> **request × capability inventory × state × risk**

低清晰 → Shape

---

## 軸 C：**Method maturity**
這題：
- 已經有成熟可重用做法？
- 還是還在 ad hoc？

高成熟 → Skill
低成熟 → Task / Shape

---

## 軸 D：**Primary posture**
這次主要是在：
- 做事（act）
- 查事（inspect）
- 建世界（world）
- 收成（harvest）

這一軸很像「使用者姿態」。

- act → Skill / Task
- inspect → Review
- world → World
- harvest → Harvest

# 所以 container selection 可以長這樣

不是 classifier，
而是這種 routing：

```text
1. 先看是否明確是長期場域
是 → World

2. 否則看 path 是否不清
是 → Shape

3. 否則看主要姿態是不是 inspect/judge
是 → Review

4. 否則看是否已有成熟 skill
是 → Skill

5. 否則看是否只是 bounded one-off job
是 → Task

6. 跑完之後，如果出現重複價值/治理價值
→ Harvest
```

這不是完美分類。
但它比硬分類器穩很多，因為：

- 先看明顯 gate
- 用的是工作條件，不是抽象語意
- 允許 transition
- Harvest 是後置，不搶前置路由
