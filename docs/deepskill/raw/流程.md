
## 2) Karvi 持有 dispatch overlay
`karvi/bindings/skills/arch-spec.dispatch.yaml`

```yaml
skillId: skill.arch-spec

dispatch:
mode: hybrid
workerClass:
- review
- research
runtimeOptions:
- codex
- opencode
handoff:
inputArtifacts:
- spec-draft
outputArtifacts:
- review-report
- patch-suggestions

```

意思是：
- 平常可以本地先跑

`thyra/bindings/skills/arch-spec.runtime.yaml`

```yaml
skillId: skill.arch-spec

- has-boundaries
- has-non-goals
- has-promotion-check

- boundary-review
- promotion-decision

- do-not-collapse-into-task-list-too-early
- do-not-mix-runtime-with-pre-dag-without-explicit-request
```

意思是：
- 這 skill 只能做 spec / thinking work
- 不能偷滑成 task list
- 不能假裝自己已經在做 runtime 設計

# 現在開始模擬一次實際使用

---

## Step 0：使用者提出需求

你說：

> 「我想把今天關於 decision engineering / pre-DAG / runtime governance 的討論，壓成一組 reviewable spec stack。」

這時候主 agent 還不知道要不要叫 `arch-spec`。
它先掃 skill registry 的 description / trigger surface。

---

## Step 1：Völva 做 routing 判斷

Völva 看見：

- 問題是模糊概念
- 想先 crystallize 成 spec
- 還不到 implementation
- 也不是 production runtime incident

所以 `routing.triggerWhen` 命中，`doNotTriggerWhen` 沒命中。
於是：

> **Völva 決定 activate `skill.arch-spec`**

這裡 Völva 做的不是執行全部事情。
它只是說：

- 這是對的 skill
- 請載入這個 skill package
- 進入這個局部工作環境

---

## Step 2：agent 載入 skill package

agent 不會一次吃完整包。
而是 progressive disclosure：

1. 先讀 `SKILL.md`
2. 再讀 `references/minimum-stack.md`
3. 再讀 `assets/spec-template.md`
4. 如果需要，再讀 `references/review-checklist.md`

這裡 skill 的作用就很像你說的 **環境**：

- 告訴 agent 現在在做哪類工作
- 什麼是好輸出
- 什麼是常見錯誤
- 可以用哪些模板
- 應該先看哪幾個檔

---

## Step 3：本地產出 draft

agent 先在當前 repo 產一份：

`docs/specs/decision-engineering-v0.md`

裡面有：

- thesis
- boundary
- non-goals
- layer split
- promotion path
- open questions

這時候是 **local run**，還沒派給別人。

同時 skill local memory 留下一筆：

```json
{
"runId": "run-001",
"inputShape": "ambiguous architecture discussion",
"output": "decision-engineering-v0.md",
"notes": ["needed stronger boundary separation between pre-DAG and runtime"]
}
```

這還只是 local log，不是 precedent。

---

## Step 4：Karvi 介入 dispatch review

因為這次 skill 的 dispatch mode 是 `hybrid`，
而且這類架構 spec 常需要 second-pass review，所以 agent 可以觸發：

> **Karvi dispatch 一個 review worker**

Karvi 會做的是：

- 挑 repo / runtime
- 選 worker class = `review`
- 組 handoff package
- 把 `decision-engineering-v0.md` 丟給另一個 worker
- 等待 `review-report`

例如 Karvi 可能派：

- repo: `C:\ai_agent\thyra`
- runtime: `codex`
- workerClass: `review`

dispatch brief 可能像：

> Review this architecture spec draft.
> Focus on boundary leaks between pre-DAG decision layer and governed runtime layer.
> Do not rewrite from scratch; return missing sections, contradictions, and promotion risks.

這時你就看到差別了：

- **Völva** 不負責「派給誰」
- **Karvi** 不負責「這個 skill 是什麼」
- Karvi 只負責：**把這個 skill 形成的工作，交給合適的 worker**

---

## Step 5：Thyra 做 runtime governance

當 review worker 開始執行時，Thyra overlay 生效。
它不是重寫 skill，而是施加 runtime 規則：

- executionMode = advisory
- externalSideEffects = false
- 必須跑 smokeChecks
- 必須經過 `boundary-review`
- 不可把 architecture draft 偷偷變成 task breakdown

所以 review worker 雖然能提 patch，
但不能：

- 直接開 PR 合併
- 直接改 production config
- 直接提升 risk permissions

Thyra 管的是：

> **這次 skill run 在這個世界裡能做到哪裡，不能做到哪裡。**

---

## Step 6：人類 checkpoint

review 回來後，主 agent整合：

- 原始 draft
- Karvi 派回來的 review report
- Thyra 檢查結果

然後給你一個 promotion decision：

> 這份 spec 是否通過 boundary review？
> 是否可從 draft 升到 sandbox / promoted？

這一步如果你點頭，才真的 promotion。

---

## Step 7：Edda 寫入 precedent

這時 Edda 不會存整份 skill object 主檔。
它只記這次事件鏈：

```yaml
- event: skill.triggered
skillId: skill.arch-spec
reason: ambiguous architecture discussion

- event: skill.run.succeeded
artifact: docs/specs/decision-engineering-v0.md

- event: skill.review.dispatched
via: karvi
runtime: codex
workerClass: review

- event: skill.gotcha.confirmed
content: "agent tends to collapse architecture work into task decomposition too early"

- event: skill.promoted
from: draft
to: sandbox
```

這就是 skill 的時間維度。

下次再遇到類似情境時，Edda 可以回來提醒：

- 這 skill 以前在什麼問題上成功
- 哪個 gotcha 是真實驗證過的
- 哪次 review 揪出過什麼結構性錯誤

# 例子：`skill.deploy-checkout-service`

場景假設：

你說：

> 把 `checkout-service` 的新版本部署到 staging，驗證通過後再 gradual rollout 到 prod。

這種事如果只是 prompt，會很可怕。

- build
- deploy
- 碰 production
- 看 metrics
- 可能 rollback
- 可能影響金流

所以這個例子最能測出四個 plane 到底是不是有必要。

# 先看這個 skill 的本體長什麼樣

## Völva 持有的 core
Völva 會定義這個 skill 的**語義本體**：

- 這 skill 是拿來做什麼的
- 什麼情況該觸發
- 什麼情況不該觸發
- 它的輸入輸出是什麼
- 基本 package 裡有哪些 runbook / scripts / gotchas

例如它會知道：

### purpose
- 問題類型：service deploy / staged release / rollback-aware rollout
- 預期結果：service 成功部署、驗證通過、無 regression
- 非目標：hotfix code authoring、incident investigation、schema redesign

### routing
- `triggerWhen`
- 使用者要求 deploy 特定 service
- 已有待部署 artifact / commit / image tag
- 需要 staging → prod rollout
- `doNotTriggerWhen`
- 其實還沒 build 好
- 只是想 debug 某個 bug
- 沒有明確 deploy target

### package
skill folder 可能有：
- `SKILL.md`
- `references/deploy-checklist.md`
- `references/common-failure-signatures.md`
- `scripts/build-and-push.sh`
- `scripts/staging-smoke-test.sh`
- `scripts/prod-rollout-checks.sh`
- `assets/deploy-report-template.md`
- `gotchas.md`

也就是說，**它不是一段說明文**。
它是一個小型 deploy 環境包。

---

# Karvi 在這裡做什麼？

Karvi 不定義 deploy 是什麼。
Karvi 定義的是：

> **這個 deploy skill 需要哪些 worker、在哪些 repo/runtimes 執行、結果怎麼收回來。**

例如 Karvi overlay 可能會寫：

- build 任務派給 `implementation worker`
- smoke test 派給 `verification worker`
- rollout monitor 派給 `ops/review worker`
- prod merge / release tag 之前需要 human approval
- timeout 30 分鐘
- 如果 staging smoke test fail，不進 prod
- 如果 prod error rate spike，觸發 rollback branch

也就是：

### Karvi 管的不是「怎麼 deploy」
而是：
- 誰來做哪一段
- 在哪裡做
- 做完怎麼接力
- 失敗怎麼升級 / 回報 / 轉派

---

# Thyra 在這裡做什麼？

Thyra 管的是這個 skill 進入真實世界後的**執行制度**。

因為 deploy 不是普通 skill。
它會碰：

- external side effects
- state transitions
- irreversible-ish changes
- rollout judgment
- rollback

所以 Thyra 會定：

### execution mode
- `restricted-active`
不是 advisory，也不是 fully autonomous destructive

### permissions
- 可以讀 dashboards
- 可以讀 deploy config
- 可以執行 staging deploy
- prod rollout 只能在 human checkpoint 之後進行
- rollback 可以在特定 guardrail 下自動觸發，或至少半自動

### verification
- staging smoke check 必須過
- error rate / latency / checkout success rate 要在閾值內
- 10% canary → 50% → 100%
- 每一步都要看 signals

### judgment rules
- 如果 payment success rate 掉超過 X%，停止 rollout
- 如果 error budget 被吃掉，回滾
- 如果 metrics inconclusive，不升下一階段
- 如果 smoke test fail，不准進 prod

所以 Thyra 管的其實是：

> **這個 deploy skill 不只是執行，而是如何在受治理的世界裡執行。**

---

# Edda 在這裡做什麼？

Edda 記的不是「deploy 指令長什麼樣」，
而是：

- 這個 skill 為什麼這樣設計
- 哪次 deploy 成功 / 失敗
- 哪個 gotcha 是真實踩出來的
- 哪個 rollback 規則後來被加上
- 哪個 verification signal 最有用
- 哪個 worker handoff 失敗過

也就是：

> **這個 skill 怎麼因為真實世界接觸而變得更穩。**

# 現在開始跑一次完整使用流程

---

## Step 0：使用者下命令

你說：

> deploy checkout-service 到 staging，驗完再上 prod

這時主 agent 不應該直接：
- 自己想一套 deploy 流程
- 然後亂跑 command

它應該先問：

> 這是不是某個已知 skill 的觸發情境？

---

## Step 1：Völva 命中 `skill.deploy-checkout-service`

Völva 看見：

- 這是 deploy 類問題
- 有明確 service
- 有 staging → prod rollout 需求
- 不是 debug / authoring / planning

所以 routing 命中。
於是主 agent 進入這個 skill package。

---

## Step 2：載入 skill package，不是只讀一段文字

agent 會逐層讀：

1. `SKILL.md`
知道主流程

2. `references/deploy-checklist.md`
知道 staging/prod checklist

3. `gotchas.md`
知道常見錯誤：
- 忘了 migrate flag
- health endpoint 綠但 checkout path 其實壞
- Stripe sandbox key / prod key 混用
- canary metrics 要看 payment success，不只看 500

4. `scripts/*.sh`
知道可以直接用哪些 helper，而不是自己胡亂寫 deploy 命令

這裡就很清楚了：

> **skill 本質上是 agent 的局部工作世界。**

不是 instruction injection。
是「你現在進到 deploy world」。

---

## Step 3：Karvi 開始拆成工作鏈

Karvi 不一定把整件事交給一個 agent。
它可能拆成三段：

### A. build + artifact check
派給 build worker

### B. staging deploy + smoke verification
派給 verification worker

### C. prod gradual rollout + monitor
派給 ops worker

所以 Karvi 可能形成這樣的派工鏈：

- Task 1: build image / verify tag
- Task 2: deploy staging / run smoke tests
- Task 3: prepare prod rollout report
- Task 4: wait for human approval
- Task 5: execute 10% canary
- Task 6: observe metrics
- Task 7: continue or rollback

這就是你提醒我的 Karvi 面。
沒有 Karvi，這 skill 只是一個本地能力；
有 Karvi，它才進入真正的組織工作流。

---

## Step 4：Thyra 開始施加 runtime guardrails

當 staging deploy worker 要真的跑 deploy 時，
Thyra overlay 生效。

這裡它會做幾件事：

### 限權
- staging 可直接 deploy
- prod 不可直接 full rollout
- 某些 destructive command 被攔住
- 某些路徑只能在指定 env 跑

### 驗證要求
- staging deploy 完不能直接宣告成功
- 必須跑 smoke tests
- 必須驗核心 transaction path
- 必須產生結構化結果

### judgment
- 如果 staging smoke test fail → 停
- 如果 staging pass 但 metrics 異常 → 停
- 如果 report 缺核心 signal → 不准 promotion 到 prod

也就是說，Thyra 在這裡像什麼？
像一個 **runtime constitution**。

---

## Step 5：staging 結果回來，Edda 不急著記大歷史

先回來的是局部結果：

- build success
- staging deploy success
- smoke test: 18/18 pass
- one warning: checkout latency up 11%
- metrics need review

這時候不是所有資訊都進 Edda 成為 precedent。
先是：
- run logs
- worker reports
- local artifacts

只有**重要、確認過、影響未來規則**的東西，才升成 Edda 事件。

這點很重要。
不然 Edda 會變垃圾桶。

---

## Step 6：進入 human checkpoint

這種 deploy skill 的關鍵就在這裡。
你不該讓 agent 在沒有制度邊界的情況下自己一路衝到 prod。

所以主 agent 整合：

- Karvi 派工結果
- staging smoke report
- Thyra guardrail judgment
- risk summary

然後回來問你：

> staging 已通過，latency 小幅上升但仍在容忍區間。
> 是否批准 10% prod canary rollout？

這就是：
- AI 高操作自治
- 人類保留制度自治

---

## Step 7：批准後，Karvi 再派 prod canary

這時 Karvi 才啟動下一段：

- ops worker 執行 10% rollout
- monitor worker 讀 dashboards
- verification worker 比對基準

Karvi 負責接力。
Thyra 負責規則。
兩者分得很清楚。

---

## Step 8：prod 出現問題，Thyra 觸發 rollback logic

假設 10% canary 後：

- checkout success rate down 3.5%
- payment timeout up
- error rate exceed threshold

這時不應該靠 agent 自己「感覺一下」。

Thyra 應該有明確 judgment rule：

- if checkout_success_delta < -2%
- or error_rate > threshold
- then stop rollout
- trigger rollback path
- generate incident summary

這時候可能是：
- 自動 rollback
- 或 rollback proposal + human confirm

重點是，這不是 prompt 決定，
而是 **runtime rule 決定**。

---

## Step 9：Edda 記下真正值得記的東西

這次事件裡，Edda 不只是記「deploy failed」。

它會記比較有價值的：

- `skill.run.failed`
- `runtime.rollback.triggered`
- `gotcha.confirmed`: healthcheck 綠不代表 checkout path 安全
- `verification.signal.promoted`: checkout success rate 必須作為 primary gate
- `promotion.blocked`: prod rollout stopped at 10%
- `patch.recommended`: add canary gate on payment timeout percentile

這種東西才是真 precedent。

下次 skill package 可能就會回寫：

### gotchas.md 新增
- 不可只依賴 health endpoint
- deploy gate 必須納入 checkout success + payment timeout
- rollout 前需確認 feature flag / dependency version alignment

這時你就看到 skill 的生命週期了。

# 這個例子最能暴露什麼？

## 1. 沒有 Völva，skill 不會有清楚邊界
agent 每次都重新發明 deploy 流程。

## 2. 沒有 Karvi，skill 不能成為組織工作
只能單 agent 本地硬幹，沒有 worker 分工、handoff、artifact 回收。

## 3. 沒有 Thyra，skill 會變危險的自動化
因為它碰 production，沒有 guardrails / judgment / checkpoint 就一定出事。

## 4. 沒有 Edda，skill 不會真正變強
因為每次 deploy 的真實經驗不會沉澱成 gotcha / precedent / promotion rule。

這比「skill 是 markdown」已經不是同一個層次。

###

你這幾個問題其實是同一個核心：

# **skill 到底是「先備能力」還是「任務中途長出來的能力」？**

答案是：**兩者都是，但不是同一時機。**

我先直接回答你的三個疑問。

---

# 1. 「我們剛剛有規劃 Völva 規劃 skill 的能力嗎？」

## 有，但目前是在**角色定位 / 邊界設計**，不是已經 fully locked implementation。

也就是說，我們剛剛談到的 Völva，比較像：

> **skill crystallization / capability authoring / semantic routing 的 plane**

它負責的不是「替所有事情自動寫 skill」，而是：

- 看出某個問題模式是不是已經值得 skill 化
- 幫現有 skill 補 semantic boundary
- 從失敗/反覆操作中抽出 gotchas
- 把模糊方法壓成 skill candidate
- 幫主 agent 判斷：這是舊 skill 可處理，還是該長新 skill

所以如果你問：
> Völva 有沒有「規劃 skill 的能力」？

我的答案是：

## **有，但應該是「提案與結晶」能力，不是無條件自動生成能力」。**

這差很多。

它應該做的是：
- propose
- draft
- patch
- refine
- boundary-check

而不是：
- 每遇到一個新任務就自動發明一個新 skill

不然 skill 爆炸。

---

# 2. 「skill 是先規劃 > 跑 > 看結果 > 修 這樣嗎？」

## 對，但我會把它講得更細一點。

不是只有四步，而是六步：

### A. 遇到問題
一個需求進來：
- deploy
- debug
- architecture spec
- legal research
- DCF
- whatever

### B. 先判斷：這是已有 skill 可處理？還是 ad hoc 問題？
這一步很重要。
不是所有問題都值得 skill 化。

### C. 如果沒有 skill，先用：
- 直接處理
- 或臨時 runbook / scratch method / step plan

### D. 跑一次，看到真實結果
這時才會知道：
- 哪些步驟是必要的
- 哪些 gotchas 真的會出現
- 哪些 verification 才有效
- 哪些資訊是本來沒想到的

### E. 從結果回收 pattern
如果這問題：
- 會重複
- 很高風險
- 很多 agent / repo 會用到
- 很值得治理

那就把它升成 **skill candidate**

### F. skill 再被反覆跑、修、升格
- 補 gotchas
- 補 scripts
- 補 verification
- 補 dispatch overlay
- 補 runtime guardrails
- 最後 promotion

所以你說的：
> 先規劃 > 跑 > 看結果 > 修

方向對，但更精確是：

> **先遇到問題 → 先處理 / 試跑 → 抽 pattern → 長成 skill candidate → 再用結果修 skill → 再 promotion**

也就是：

# **skill 通常不是起點，而是從任務中結晶出來的可重用能力。**

---

# 3. 「新問題沒 skill，就要先做新 skill 嗎？」

## 不一定。這是最重要的地方。

我會直接反對「每個新問題先做 skill」。

因為那樣會讓 skill 變成：
- prompt 墳場
- 半成品倉庫
- 沒跑過的抽象模板

那很糟。

---

## 我會用這個判斷準則：

### 不要先做 skill 的情況
如果問題是：
- 一次性的
- 還不確定問題形狀
- 還沒跑過
- 還不知道真正 gotcha 在哪
- 還不知道驗證怎麼做

那就**先不要 skill 化**。

先：
- ad hoc 處理
- 臨時 step plan
- 臨時派工
- 收集真實失敗與結果

---

### 應該早點 skill 化的情況
如果問題是：
- 高風險
- 會反覆出現
- 多人/多 agent 都會做
- 需要穩定 handoff
- 需要 verification
- 需要治理與審核
- 出錯代價高

那就應該比較早地做 **最小 skill skeleton**。

例如：
- deploy
- oncall/debugging
- dependency management
- prod maintenance
- sensitive review workflows

這些不是等跑十次才開始 skill 化。
這種通常一開始就該有最小骨架：

- trigger
- checklist
- permissions
- verification
- human checkpoint

因為風險太高。

---

# 你問 deploy 問題特別好，因為它剛好卡在中間

## deploy 這類問題不是「完全不能直接派發」
可以直接派發。

但如果你問：

> 沒有 skill 的 direct dispatch
> vs
> 有最小 skill skeleton 的 dispatch

哪個比較好？

我的答案很明確：

# **高風險任務，用 skill + dispatch 幾乎一定比純 dispatch 好。**

因為 pure dispatch 只有：
- 任務描述
- worker 自由發揮

但 skill + dispatch 會多出：

- 明確 trigger 與邊界
- 常見 gotchas
- 驗證腳本
- 風險限制
- human checkpoint
- 歷史可累積

這就是為什麼 deploy / oncall / maintenance 這些，我會偏向：

> **不要裸派工。要用至少最小 skill skeleton 再派。**

---

# 那是不是「高階任務一律 skill + step」？

## 我會說：**不是一律，但大多數高價值任務會往這個方向收斂。**

更準的說法不是 skill + step，
而是：

# **route → capability → plan/steps → execution**

也就是四層：

### 1. Route
先判斷這是什麼類問題

### 2. Capability
看有沒有現成 skill / capability object

### 3. Plan / Steps
針對這次具體任務，做一次性的 step decomposition

### 4. Execution
本地跑或透過 Karvi 派工跑

---

## skill 跟 steps 不是同一件事

這很重要。

### Skill
是**可重用能力包**
- reusable
- governable
- reusable semantics
- scripts / gotchas / verification

### Steps / plan
是**這次任務的具體拆解**
- one-run specific
- depends on current context
- repo/runtime/time sensitive

所以不是：
- skill 取代 steps

而是：
- **skill 提供處理這類問題的能力框架**
- **steps 是這次執行的局部行動序列**

---

# 我給你一個更實用的 decision table

## Case 1：低風險、一次性、邊界清楚
例如：
- 改一個 function
- 補 test
- rename 一批檔案

### 做法
直接 task / plan / dispatch
**不用 skill**

---

## Case 2：中風險、可能重複、已有些 pattern
例如：
- architecture spec
- recurring analysis workflow
- DCF 類分析流程

### 做法
可先用 **skill candidate + steps**
skill 還在 sandbox，不必太重

---

## Case 3：高風險、會重複、需要治理
例如：
- deploy
- prod debug
- maintenance
- dependency approval
- cost investigation
- compliance-ish workflows

### 做法
應該要 **skill skeleton + steps + dispatch + runtime guardrails**
這時 skill 幾乎是必須的

---

# 所以 Völva 的判斷流程，應該長這樣

不是：
> 新問題 → 建 skill

而是：

> 新問題
> → 判斷是否已有 skill
> → 若無，判斷先 ad hoc 處理還是先建最小 skill skeleton
> → 跑一次
> → 看結果
> → 若值得重用，再 crystallize 成正式 skill

這個比較對。

# 所以最短答案是：

## Q1. Völva 有規劃 skill 的能力嗎？
有，但應該是**提出、結晶、修補、邊界化**，不是每次自動新建。

## Q2. skill 是先規劃 > 跑 > 看結果 > 修嗎？
大致是，但更準是：**先跑問題，再從結果長 skill。**

## Q3. 新問題一定先做 skill 嗎？
不一定。
**低風險/一次性 → 直接處理。**
**高風險/可重複/需治理 → 先做最小 skill skeleton 再處理更好。**

## Q4. 高階任務可以一律 skill + step 嗎？
不該一律。
但很多高價值、高風險、高重複任務，最後會自然收斂成：

> **skill 提供能力框架，steps 提供本次任務拆解。**
